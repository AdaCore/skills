# Corpora and Test Inputs

A **corpus** is a directory of binary files; each file is a serialized test
input to the target subprogram. `gnatfuzz` and AFL++ refer to the corpus
also as a "queue" ŌĆö the queue of pending or processed test cases.

Corpus files are **binary**, encoded by **TGen** (Test Generator)
marshalling. TGen lays parameters out compactly without the padding a C-style
struct dump would have, which both makes file sizes small and increases the
probability that random mutation produces another valid input.

You will work with corpora in three forms:

1. **The starting corpus** ŌĆö what you give to `gnatfuzz fuzz` via
   `--corpus-path`. Auto-generated by `gnatfuzz generate-corpus`,
   optionally augmented or replaced by `User_Provided_Seeds`, optionally
   minimized by `gnatfuzz minimize-corpus`.
2. **The runtime queue** ŌĆö `<harness>/fuzz_testing/session/.../queue/`,
   populated by AFL++ or libFuzzer during the campaign. New files appear here
   when the fuzzer discovers an input that exercises a new path. Treat this as
   read-only during the campaign; it's an artifact, not an input.
3. **The ending corpus** ŌĆö `<harness>/fuzz_testing/session/results/ending_corpus/`
   after the campaign ends. Useful as the seed for a future campaign.

## Generating a starting corpus

```bash
gnatfuzz generate-corpus -P <harness>/fuzz_test.gpr -o <harness>/starting_corpus
```

The `--strategy` flag (default `internal,user`) controls how:

* `internal` ŌĆö sample each parameter type at boundary and middle values.
  For `Integer`-family types this is `T'First`, a midpoint, and `T'Last`.
  For arrays of integral types (including `String`), generate combinations
  of element values. For records, combine per-field samples.
* `user` ŌĆö call `User_Provided_Seeds` in
  `<harness>/user_configuration/user_configuration.adb` and add every seed
  it produces.

`internal` works for:

* Discrete integer types (`Integer`, `Natural`, `Positive`, custom
  `range`-constrained subtypes).
* Arrays of integral types, including `String`.
* Records composed of the above.

`internal` does **not** work for subprograms with generic formal parameters
or anything that lands in `non_fuzzable_subprograms`. For those, use
`--strategy user` and provide seeds manually (next section), or use a
**dummy corpus** (random bytes that TGen marshalling will progressively
mutate into valid values during the campaign).

User-provided seeds are named `User_1`, `User_2`, etc. Seeds that come from
automatic corpus generation are roughly the fully qualified name plus a hash
that is calculated by TGen (in order to handle function overloads).

## Manually providing seeds

`gnatfuzz generate` writes a stub `User_Provided_Seeds` procedure into
`<harness>/user_configuration/user_configuration.adb`. Edit it, then
rebuild the harness:

```ada
procedure User_Provided_Seeds (Add_Corpus_Seed : Corpus_Seed_Callback) is
begin
   --  One call per test case. Parameter names match the target subprogram.
   Add_Corpus_Seed (Fuzz_Input_Date => (Day => 24, Month => February, Year => 1978));
   Add_Corpus_Seed (Fuzz_Input_Date => (Day => 29, Month => March,    Year => 1984));
   --  ...
end User_Provided_Seeds;
```

After editing, rebuild and regenerate the corpus:

```bash
gnatfuzz build           -P <harness>/fuzz_test.gpr
gnatfuzz generate-corpus -P <harness>/fuzz_test.gpr --strategy user -o <harness>/starting_corpus
```

`--strategy user` skips internal sampling ŌĆö useful when the parameter
types defeat internal generation, or when you have a high-quality manual
seed set you don't want diluted with boundary values.

## Reading a corpus entry

Corpus files are binary; opening one in a text editor will show garbage.
The harness build produces a **decoder** binary under
`<harness>/build/obj-AFL_PERSIST/` that prints the decoded parameter values
of any file in the corpus directory.

Replay/decode a corpus entry the same way you replay a crash:

```bash
<harness>/build/obj-AFL_PERSIST/gnatfuzz_test_harness.verbose <path-to-corpus-file>
```

The output is JSON with `Decoded_In_Parameters` (the parameter names,
types, and values), `Subprogram_Under_Test`, and ŌĆö for files that triggered
an exception ŌĆö `Testcase_Exception` with the exception name, message, and
location. See [outputs.md ┬¦ Replaying a crash](../outputs/outputs.md#replaying-a-crash)
for the full output schema.

The same binary works for queue files, crash files, hang files, and
starting-corpus files: they all use the same TGen marshalling format.

## Minimizing a corpus

```bash
gnatfuzz minimize-corpus -P <harness>/fuzz_test.gpr \
    -i <harness>/starting_corpus \
    -o <harness>/starting_corpus_min
```

Minimization runs `afl-cmin` (for AFL++/CMPLOG) or libFuzzer's merge mode
to keep only the inputs that exercise unique execution paths. The output
is typically much smaller than the input ŌĆö internal generation often
produces redundant inputs that all hit the same path.

`gnatfuzz fuzz` minimizes the starting corpus automatically before the
campaign begins. Calling `minimize-corpus` explicitly is useful when:

* You want to check the minimized corpus into version control as the
  durable starting set for repeat campaigns.
* You're going to run multiple campaigns from the same starting corpus and
  don't want to pay minimization overhead each time.

The user's guide documents `--no-cmin` to skip pre-campaign minimization
on `gnatfuzz fuzz`. Verify with `gnatfuzz fuzz --help` for your release.

## TGen, JSON test vectors, and incremental campaigns

For incremental fuzzing across multiple campaigns, the beta
`--tgen-gnattest-test-generator` flag on `gnatfuzz fuzz` watches the AFL
queue and emits each new test case into a TGen JSON file. The
`gnatfuzz import-tgen-vectors` utility then re-imports those vectors as
seeds the next time you run `generate-corpus`. This lets a follow-on
campaign start from where the previous one ended ŌĆö useful in CI where
campaigns run for fixed wall-clock budgets.

## Anti-patterns

* **Hand-editing a corpus file.** The format is binary TGen marshalling;
  there is no human-editable form. Add seeds via `User_Provided_Seeds`
  and rebuild.

* **Treating the runtime `queue/` as your starting corpus for the next
  campaign.** It works, but `results/ending_corpus/` is the curated
  artifact `gnatfuzz` produces for that purpose.

* **Skipping the starting corpus entirely.** The fuzzer can be fed an
  empty directory, but it will spend a long time discovering valid inputs
  from random bytes. Always seed something ŌĆö even a single `GNATfuzz_1`
  from `--strategy internal` is far better than nothing.

* **Forgetting to rebuild after editing `User_Provided_Seeds`.** The
  procedure is compiled into the harness; without `gnatfuzz build`, your
  edits are not used.
